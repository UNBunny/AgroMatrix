# AgroMatrix

AgroMatrix — информационная система для поддержки аграрного планирования и управления сельскохозяйственным хозяйством.

Проект предназначен для работы с полями, культурами, погодными данными, спутниковыми показателями NDVI, агрономическими рекомендациями, прогнозами урожайности и отчётностью.

## Назначение проекта

Система разработана для автоматизации части процессов, связанных с управлением сельскохозяйственными полями и принятием агрономических решений.

AgroMatrix помогает хранить данные о хозяйствах и полях, анализировать погодные условия, отслеживать состояние растительности по NDVI, оценивать риски заболеваний растений и получать рекомендации для планирования сельскохозяйственных работ.

## Актуальность

В сельском хозяйстве часто используются разрозненные источники данных: сведения о полях, погодные сервисы, спутниковые снимки, таблицы с культурами, ручные отчёты и отдельные расчёты. Это усложняет анализ состояния полей и принятие решений.

AgroMatrix объединяет основные данные в одной системе и предоставляет пользователю инструменты для анализа, мониторинга и планирования. Это снижает объём ручной работы и повышает удобство управления аграрными процессами.

## Цель проекта

Цель проекта — разработать веб-систему для управления сельскохозяйственными полями и поддержки принятия решений в аграрном планировании на основе погодных данных, NDVI, справочной информации и моделей машинного обучения.

## Основные задачи

- реализовать регистрацию пользователей и хозяйств;
- обеспечить управление сельскохозяйственными полями;
- реализовать работу с культурами, сортами и севооборотом;
- добавить анализ погодных и агрометеорологических данных;
- реализовать мониторинг NDVI для оценки состояния растительности;
- добавить оценку рисков заболеваний растений;
- реализовать рекомендации по культурам и аграрным действиям;
- добавить прогнозирование урожайности и рыночных цен;
- реализовать отчёты, планы и журнал аудита;
- разработать административную панель;
- обеспечить взаимодействие сервисов через API Gateway и внутренние интеграции.

## Основные возможности

### Управление пользователями и хозяйствами

В системе предусмотрена авторизация пользователей, регистрация хозяйств, управление ролями и привязка пользователей к хозяйствам.

### Управление полями

Пользователь может создавать и редактировать поля, хранить сведения о площади, координатах, почвенных параметрах и связанных культурах.

### Работа с культурами

Система поддерживает справочники культур, сортов, историю посевов, севооборот и агрономические рекомендации.

### Погодный анализ

Weather Service получает и обрабатывает погодные данные. Эти данные используются для расчёта агрометеорологических показателей, анализа условий выращивания и оценки рисков.

### Мониторинг NDVI

NDVI Service отвечает за обработку данных спутникового мониторинга. Показатель NDVI используется для оценки состояния растительности на поле.

### Оценка рисков заболеваний

Система использует погодные и агрономические данные для оценки риска заболеваний растений. Результат отображается пользователю в виде уровня риска и пояснения.

### ML-рекомендации и прогнозы

Agro ML Service содержит модели и API для прогнозирования урожайности, цен, рекомендаций по культурам, удобрениям и средствам защиты растений.

### Отчёты и планы

В системе реализованы отчёты по полям, планы сезона, согласование планов и журнал аудита действий.

### Административная панель

Административный интерфейс позволяет управлять пользователями, справочниками и основными данными системы.

## Архитектура системы

Проект построен по микросервисной архитектуре. Каждый сервис отвечает за отдельную часть предметной области.

### Состав системы

- `frontend` — пользовательский интерфейс на React;
- `api-gateway` — единая точка входа для frontend и маршрутизация запросов;
- `auth-service` — авторизация, регистрация, роли пользователей и JWT;
- `agriculture-field-service` — хозяйства, поля, культуры, севооборот, отчёты и планы;
- `weather-service` — получение и обработка погодных данных;
- `ndvi-service` — мониторинг NDVI и обработка спутниковых данных;
- `agro-ml-service` — ML-сервис для рекомендаций и прогнозов;
- `eureka-server` — регистрация и обнаружение сервисов.

## ML-сервис

ML-сервис — это отдельный компонент системы, построенный на Python и FastAPI. Он выполняет прогнозирование сельскохозяйственных показателей с использованием моделей машинного обучения.

### Функции ML-сервиса

| Функция | Описание | Модель |
|---------|----------|--------|
| Прогноз урожайности | Предсказание урожайности по региону, культуре и погодным данным | LightGBM |
| Прогноз цен | Прогнозирование рыночных цен на сельскохозяйственную продукцию | LightGBM |
| Рекомендации культур | Подбор оптимальной культуры по параметрам почвы и климата | scikit-learn |
| Рекомендации удобрений | Подбор удобрений по типу культуры и дефициту питательных веществ | scikit-learn |
| Рекомендации пестицидов | Подбор средств защиты по типу вредителя и степени заражения | scikit-learn |
| Оценка рисков болезней | Прогнозирование риска заболеваний растений по погодным условиям | scikit-learn |
| Анализ угроз и защита | Комплексный анализ фитосанитарных угроз с рекомендациями препаратов | Правила + ML |
| NDVI-аналитика | Получение спутниковых данных NDVI через Google Earth Engine | GEE API |

Проект состоит из нескольких сервисов, поэтому запуск выполняется поэтапно. Сначала поднимается инфраструктура, затем сервисы backend, ML-сервис и frontend.

Для запуска проекта нужны:

- Java 21;
- Maven или Maven Wrapper из проекта;
- Node.js 18 или выше;
- Python 3.11;
- PostgreSQL с расширением PostGIS;
- Redis;
- Kafka;
- Docker, если инфраструктура запускается через контейнеры.

### Подготовка инфраструктуры

Перед запуском сервисов должны быть доступны:

- PostgreSQL на `localhost:5432`;
- Redis на `localhost:6379`;
- Kafka на `localhost:9092`.

Для PostgreSQL используются стандартные параметры локального запуска:

```text
host: localhost
port: 5432
user: admin
password: admin
```

Необходимо создать базы данных:

```text
agro_plan

```

Для запуска через Docker Compose в проекте используется скрипт:

```text
docker/postgres/init-multiple-databases.sh
```

Он нужен для создания нескольких баз данных при запуске PostgreSQL в контейнере.

### Запуск через Docker Compose

Docker Compose позволяет запустить инфраструктуру и основные сервисы одной командой.

Перед запуском нужно убедиться, что Docker Desktop запущен.

Запуск всех контейнеров:

```powershell
docker compose up --build
```

Если нужно запустить контейнеры в фоновом режиме:

```powershell
docker compose up --build -d
```

После запуска будут подняты:

- PostgreSQL с базами `agro_plan` и `agro_auth`;
- Redis;
- Zookeeper;
- Kafka;
- Eureka Server;
- API Gateway;
- Auth Service;
- Agriculture Field Service;
- Weather Service;
- NDVI Service;
- Agro ML Service.

Проверить состояние контейнеров:

```powershell
docker compose ps
```

Посмотреть логи всех сервисов:

```powershell
docker compose logs -f
```

Посмотреть логи отдельного сервиса:

```powershell
docker compose logs -f api-gateway
```

Остановить контейнеры:

```powershell
docker compose down
```

Остановить контейнеры и удалить volume с данными PostgreSQL:

```powershell
docker compose down -v
```

Команду с `-v` нужно использовать осторожно, потому что она удалит данные базы.

После запуска через Docker Compose основные адреса остаются такими же:

- frontend запускается отдельно на `http://localhost:5173`;
- API Gateway доступен на `http://localhost:8080`;
- Eureka Server доступен на `http://localhost:8761`;
- ML Service доступен на `http://localhost:8090`;
- Swagger ML Service доступен на `http://localhost:8090/docs`.

Frontend в текущем `docker-compose.yml` не запускается, поэтому его нужно запускать отдельно:

```powershell
cd frontend
npm install
npm run dev
```

В `docker-compose.yml` для ML-сервиса указан локальный путь к Google Earth Engine credentials:

```text
C:/Users/nikit/.config/earthengine:/root/.config/earthengine:ro
```

Если проект запускается на другом компьютере, этот путь нужно заменить на путь к своим credentials или убрать volume, если функции Google Earth Engine не используются.

В системе предусмотрены разные сценарии работы пользователей. Основными ролями являются пользователь хозяйства и администратор.

Пользователь хозяйства работает с полями, культурами, рекомендациями, отчётами и планами. Администратор управляет пользователями и справочными данными.

## Данные, используемые системой

Система использует несколько групп данных:

- данные пользователей и хозяйств;
- данные о сельскохозяйственных полях;
- координаты и геометрия полей;
- сведения о культурах и сортах;
- погодные данные;
- спутниковые показатели NDVI;
- данные для ML-моделей;
- отчёты, планы и записи аудита.

## Практическая значимость

AgroMatrix может использоваться как основа для цифрового инструмента аграрного планирования. Система объединяет управление полями, погодный анализ, спутниковый мониторинг и ML-рекомендации в одном интерфейсе.

Такой подход позволяет быстрее получать информацию о состоянии полей, планировать сельскохозяйственные работы и принимать более обоснованные решения.

## Результат разработки

В результате разработки создана веб-система, включающая frontend, набор backend-сервисов, ML-сервис, базу данных, кэширование и механизмы взаимодействия между сервисами.

Система демонстрирует применение микросервисной архитектуры, REST API, геоданных, спутникового мониторинга, машинного обучения и современных средств веб-разработки для решения задач аграрного планирования.
